# GARC: Follow-Up RC Soak

## Context

GARC is the sixth phase in the v5 GA-hardening roadmap and the first phase
that is allowed to dispatch release automation. `GABASE`, `GAGOV`,
`GASUPPORT`, `GAE2E`, and `GAOPS` are already complete in
`.codex/phase-loop/state.json`, while `GARC` remains `unplanned`.

Current repo state gathered during planning:

- The checkout is on `main` at `8d08545c15c53322128ef87b5e06308bd8b0dad3`.
- `specs/phase-plans-v5.md` is already staged in this worktree
  (`git status --short -- specs/phase-plans-v5.md` returned
  `A  specs/phase-plans-v5.md`), so it must be treated as the active
  user-owned roadmap baseline rather than rewritten here.
- `docs/validation/ga-readiness-checklist.md`,
  `docs/validation/ga-governance-evidence.md`,
  `docs/validation/ga-e2e-evidence.md`, and
  `docs/validation/ga-operations-evidence.md` already exist and freeze the
  upstream contract that GARC must consume.
- The checklist and operator runbooks already freeze `v1.2.0-rc6` as the
  follow-up RC target, but the executable release surfaces are still frozen to
  `v1.2.0-rc5`: `.github/workflows/release-automation.yml`,
  `pyproject.toml`, `mcp_server/__init__.py`, `tests/test_release_metadata.py`,
  the installer helpers, and the customer-facing install docs still point at
  `rc5`.
- `docs/validation/ga-rc-evidence.md` does not exist yet and is the canonical
  new artifact required by this phase.
- `RELEASE_NOTES.md` is not currently tracked in the repo and is generated by
  the release workflow during `Prepare Release`; GARC should treat it as a
  workflow-produced artifact to observe and record, not as a source file to
  hand-maintain before dispatch.
- The existing RC5 evidence in `docs/validation/rc5-release-evidence.md`
  proves the release workflow can publish a prerelease, but it also shows that
  tag creation, GitHub release publication, PyPI publication, and GHCR publish
  are tightly coupled inside `.github/workflows/release-automation.yml`; GARC
  must therefore freeze the `rc6` contract surfaces before dispatch rather than
  relying on the old `rc5` defaults.

## Interface Freeze Gates

- [ ] IF-0-GARC-1 - Follow-up RC contract surface:
      `.github/workflows/release-automation.yml`, `pyproject.toml`,
      `mcp_server/__init__.py`, `tests/test_release_metadata.py`, installer
      defaults, and public install/version docs all freeze
      `v1.2.0-rc6` / `1.2.0-rc6` as the active follow-up RC while preserving
      prerelease-only posture.
- [ ] IF-0-GARC-2 - Pre-dispatch qualification contract:
      local execution requires a clean expected branch, synchronized
      `origin/main`, no reused `v1.2.0-rc6` tag locally or remotely, and fresh
      upstream GAGOV, GAE2E, and GAOPS evidence before `gh workflow run`
      executes.
- [ ] IF-0-GARC-3 - Workflow observation contract:
      GARC records the Release Automation run URL, run ID, `headSha`,
      per-job conclusions, release branch / PR disposition, GitHub release
      state, tag target, PyPI publication, and GHCR image identity for
      `v1.2.0-rc6`.
- [ ] IF-0-GARC-4 - RC-only channel policy contract:
      the follow-up RC remains prerelease/public-alpha or beta posture,
      `auto_merge=false` remains the default unless a fresh governance decision
      explicitly approves otherwise, Docker `latest` remains stable-only, and
      no GA wording or stable-channel claims are introduced by the `rc6` soak.
- [ ] IF-0-GARC-5 - Evidence reducer contract:
      `docs/validation/ga-rc-evidence.md` is the canonical GARC artifact and
      records workflow URLs, artifact identities, tag target, release-channel
      state, rollback disposition, and any manual decisions or blocked-stop
      reasons using redacted metadata only.
