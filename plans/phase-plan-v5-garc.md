# GARC: Follow-Up RC Soak

## Context

GARC is the sixth phase in the v5 GA-hardening roadmap and the first phase
that is allowed to dispatch release automation. `GABASE`, `GAGOV`,
`GASUPPORT`, `GAE2E`, and `GAOPS` are already complete, and the repository has
already landed most of the `rc6` contract-freeze work that this phase depends
on. The remaining GARC-critical work is to re-qualify the release state on the
current checkout, dispatch the follow-up RC only from a synchronized `main`,
and refresh the canonical GARC evidence from the actual workflow outcome.

Current repo state gathered during planning:

- The checkout is on `main` at `96998dd66de837b64914192e4967ee80fbe34e55`
  with a clean worktree, but `origin/main` is still
  `8d08545c15c53322128ef87b5e06308bd8b0dad3`, so the branch-sync portion of the
  GARC pre-dispatch contract is not currently satisfied.
- `specs/phase-plans-v5.md` is tracked and clean in this worktree, so it
  remains the active roadmap baseline for this refreshed plan.
- `docs/validation/ga-readiness-checklist.md`,
  `docs/validation/ga-governance-evidence.md`,
  `docs/validation/ga-e2e-evidence.md`, and
  `docs/validation/ga-operations-evidence.md` already exist and freeze the
  upstream contract that GARC must consume.
- `.github/workflows/release-automation.yml`, `pyproject.toml`,
  `mcp_server/__init__.py`, `tests/test_release_metadata.py`, `CHANGELOG.md`,
  the active install docs, installer helpers, and the GARC runbooks already
  freeze `v1.2.0-rc6` / `1.2.0-rc6`, so GARC execution should now treat those
  surfaces as verification-first and repair-only-on-drift.
- `docs/validation/ga-rc-evidence.md` already exists as the canonical GARC
  artifact and currently records a historical `blocked before dispatch`
  attempt; successful execution must update that artifact with the current
  qualification and observation results instead of creating a parallel record.
- `RELEASE_NOTES.md` is not currently tracked in the repo and is generated by
  the release workflow during `Prepare Release`; GARC should treat it as a
  workflow-produced artifact to observe and record, not as a source file to
  hand-maintain before dispatch.
- `docs/validation/rc5-release-evidence.md` still serves as the last published
  prerelease proof, while `docs/validation/ga-rc-evidence.md` shows that the
  first `rc6` attempt stopped before dispatch on a dirty worktree at commit
  `8d08545c15c53322128ef87b5e06308bd8b0dad3`; this refresh supersedes that
  stale blocker with the current branch-sync gate and preserves the same
  canonical evidence path.

      GARC records the Release Automation run URL, run ID, `headSha`,

      the follow-up RC remains prerelease/public-alpha or beta posture,
      `auto_merge=false` remains the default unless a fresh governance decision
      explicitly approves otherwise, Docker `latest` remains stable-only, and
      no GA wording or stable-channel claims are introduced by the `rc6` soak.

      `docs/validation/ga-rc-evidence.md` is the canonical GARC artifact and
      records workflow URLs, artifact identities, tag target, release-channel
      state, rollback disposition, and any manual decisions or blocked-stop
      reasons using redacted metadata only.
