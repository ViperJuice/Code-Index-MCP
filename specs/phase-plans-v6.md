# Phase roadmap v6

## Context

The v5 roadmap completed the GA release track and published stable `v1.2.0`.
This roadmap starts a separate multi-repo production-hardening family. It is
not a rollback of the GA decision and must not reopen v5 release governance.

The review that triggered this roadmap found that the current multi-repo
architecture is close, but several success signals are too optimistic for
hands-off deployment across many repositories. The most important gaps are
remote artifact durability, commit advancement after partial indexing failure,
multi-repo watcher event truth, GitHub artifact workflow parity, and end-to-end
proof that published artifacts can hydrate clean workspaces for multiple repos.

## Architecture North Star

Multi-repo deployment should be boring and fail-closed: every registered
repository has one tracked/default branch, one durable local index, explicit
readiness, and a remote artifact path that can hydrate a clean workspace without
manual repair. Index state may only advance when mutations have actually landed.
Remote pointers may only advance after the referenced artifact assets are
uploaded and validated.

## Assumptions

- `v1.2.0` is the current stable release baseline.
- The v3 topology contract remains in force: many unrelated repositories, one
  registered worktree per git common directory, tracked/default branch indexing,
  and fail-closed readiness for unavailable or unsupported indexes.
- GitHub Releases and GitHub Actions remain the artifact publication surfaces.
- `.mcp-index/current.db` remains the canonical local lexical index.
- Artifact metadata must continue to carry the P31 identity fields: `repo_id`,
  `tracked_branch`, `commit`, schema, semantic profile hash, checksum, and
  manifest v2.
- No phase in this roadmap authorizes unrestricted per-branch remote artifact
  publishing.

## Non-Goals

- No new stable package release unless a downstream release roadmap explicitly
  asks for one.
- No multi-worktree support expansion beyond the existing fail-closed model.
- No remote delta-download protocol.
- No S3, GCS, or Azure production provider enablement.
- No broad plugin or parser rewrite.
- No marketing or launch copy.

## Cross-Cutting Principles

- Prefer explicit mutation results over log-only failure handling.
- Treat `ready` as a durable state claim, not a best-effort status.
- Keep local-first behavior available when remote artifact publication is not
  configured.
- Keep workflow metadata generation and runtime metadata generation aligned.
- Add tests that exercise the real failure modes, not only optimistic mocks.
- Keep docs honest about supported, beta, and local-only paths.

## Top Interface-Freeze Gates

- IF-0-ARTPUB-1 — Remote artifact publish contract: publish-on-reindex writes
  metadata, uploads archive assets, validates the SHA release, then moves the
  latest pointer only after durable assets exist.
- IF-0-IDXSAFE-1 — Index mutation truth contract: add, modify, delete, and move
  operations expose explicit success/failure results, and commit advancement is
  blocked on any failed required mutation.
- IF-0-WATCH-1 — Watcher event truth contract: multi-repo live events mark a
  repo changed only after a real mutation, explicit skip, or queued safe
  recovery path is recorded.
- IF-0-CIFLOW-1 — CI artifact workflow parity contract: reusable workflow
  upload, push-trigger upload, and direct publish generate compatible archive
  and metadata assets.
- IF-0-MRE2E-1 — Multi-repo hydration evidence contract: a clean workspace can
  hydrate published artifacts for multiple repos, reject wrong-branch state,
  and return correct search/readiness results after add, modify, delete, and
  rename changes.
- IF-0-MRREADY-1 — Multi-repo rollout gate contract: docs, CLI status, and
  operator runbooks distinguish ready, local-only, wrong-branch,
  publish-failed, partial-failure, and unavailable states.

## Phases

### Phase 1 — Remote Artifact Publish Durability (ARTPUB)

**Objective**

Make `ArtifactPublisher.publish_on_reindex()` produce durable, downloadable
release assets instead of only release pointers.

**Exit criteria**
- [ ] The SHA-keyed release contains the compressed index archive,
      `artifact-metadata.json`, checksum material, and attestation material when
      those assets are available.
- [ ] Metadata generated by `create_metadata()` is persisted and matches the
      P31 artifact identity contract.
- [ ] `index-latest` is created or moved only after the SHA-keyed release has
      the required assets.
- [ ] Repeated publish calls for the same repo/branch/commit remain idempotent.
- [ ] Race tests assert asset upload behavior, not only release create/edit
      ordering.
- [ ] Download validation can consume an artifact produced by the direct
      publish path.

**Scope notes**

This phase owns the runtime direct-publish path and its tests. It may reuse the
existing uploader helper if that keeps the release and metadata behavior
consistent.

**Non-goals**

- No new remote provider.
- No remote delta-download protocol.
- No watcher behavior changes except through the publisher interface.

**Key files**

- `mcp_server/artifacts/publisher.py`
- `mcp_server/artifacts/artifact_upload.py`
- `mcp_server/artifacts/artifact_download.py`
- `mcp_server/artifacts/attestation.py`
- `docs/INDEX_MANIFEST_CONTRACT.md`
- `docs/guides/artifact-persistence.md`
- `tests/test_artifact_publish_race.py`
- `tests/test_artifact_auto_delta.py`
- `tests/test_artifact_upload.py`
- `tests/test_artifact_download.py`

**Depends on**
- (none)

**Produces**
- IF-0-ARTPUB-1 — Remote artifact publish contract.

### Phase 2 — Index Commit Advancement Safety (IDXSAFE)

**Objective**

Ensure a repository commit is recorded as indexed only after every required
file-level mutation succeeds or an explicit safe recovery path is selected.

**Exit criteria**
- [ ] Dispatcher add/modify/delete/move paths expose explicit mutation outcomes
      or raise consistently on failed required mutations.
- [ ] `GitAwareIndexManager._incremental_index_update()` treats skipped,
      failed, or unknown required mutations as non-clean.
- [ ] `last_indexed_commit` is not advanced after any partial incremental
      failure.
- [ ] Delete and move accounting verifies that the intended store mutation
      actually affected the indexed path when that path was expected to exist.
- [ ] Tests cover the real dispatcher behavior where errors may otherwise be
      logged and swallowed.
- [ ] Existing wrong-branch and missing-index fail-closed behavior remains
      unchanged.

**Scope notes**

This phase freezes the mutation-result contract before watcher hardening relies
on it. The implementation should keep compatibility with ctx-aware dispatcher
calls and avoid broad dispatcher rewrites.

**Non-goals**

- No new query surface.
- No support for indexing non-tracked branches.
- No full storage-engine replacement.

**Key files**

- `mcp_server/storage/git_index_manager.py`
- `mcp_server/dispatcher/dispatcher_enhanced.py`
- `mcp_server/storage/sqlite_store.py`
- `mcp_server/core/repo_context.py`
- `mcp_server/storage/repository_registry.py`
- `tests/test_git_index_manager.py`
- `tests/test_incremental_indexer.py`

**Depends on**

**Produces**
- IF-0-IDXSAFE-1 — Index mutation truth contract.

### Phase 3 — Multi-Repo Watcher Event Correctness (WATCH)

**Objective**

Make live multi-repo file events report truthfully and recover safely from
branch, gitignore, TOCTOU, and rename/delete edge cases.

**Exit criteria**
- [ ] Multi-repo reindex events honor `IndexResult` from guarded indexing.
- [ ] Repositories are marked changed only when a mutation landed or an
      explicit recovery action was queued.
- [ ] Move events verify the destination still exists before recording a move,
      matching the single-repo watcher behavior.
- [ ] Successful remote publish updates artifact health away from stale
      `local_only` state, while failures remain visible as `publish_failed`.
- [ ] `sync_all_repositories(parallel=True)` semantics are either implemented
      where advertised or the misleading argument is removed from the manager
      API.
- [ ] Tests cover two watched repositories, wrong-branch drops, guarded-index
      skips, move-destination disappearance, and publish health transitions.

**Scope notes**

This phase depends on the mutation-result contract from IDXSAFE. It may adjust
status bookkeeping, but should not broaden the product topology.

**Non-goals**

- No unrestricted branch watcher.
- No remote provider changes.
- No new CLI command unless needed to expose existing status truth.

**Key files**

- `mcp_server/watcher_multi_repo.py`
- `mcp_server/watcher/file_watcher.py`
- `mcp_server/watcher/ref_poller.py`
- `mcp_server/storage/git_index_manager.py`

**Produces**
- IF-0-WATCH-1 — Watcher event truth contract.

### Phase 4 — CI Artifact Workflow Parity (CIFLOW)

**Objective**

Repair and align GitHub artifact workflows so CI uploads and runtime direct
publishes produce compatible artifact payloads and metadata.

**Exit criteria**
- [ ] The reusable `index-artifact-management.yml` upload path has valid shell
      and Python syntax when `.index_metadata.json` is present.
- [ ] Workflow-dispatch inputs do not advertise unsupported actions.
- [ ] Push-trigger upload, reusable workflow upload, and direct runtime publish
      produce compatible `artifact-metadata.json` fields.
- [ ] Workflow tests or syntax checks exercise metadata-present upload.
- [ ] Docs name which path operators should use for local-only, CI, and
      publish-on-reindex workflows.

**Scope notes**

This phase owns GitHub Actions and scripts only. Runtime publisher behavior is
owned by ARTPUB; this phase keeps workflow behavior compatible with that
contract.

**Non-goals**

- No stable package release dispatch.
- No repository settings mutation.
- No new artifact provider.

**Key files**

- `.github/workflows/index-artifact-management.yml`

**Depends on**

**Produces**
- IF-0-CIFLOW-1 — CI artifact workflow parity contract.

### Phase 5 — Multi-Repo Hydration Evidence (MRE2E)

**Objective**

Prove the complete multi-repo deployment model from fresh state: register,
index, publish, hydrate, reconcile, and query across multiple repositories.

**Exit criteria**
- [ ] An end-to-end test registers at least two unrelated repositories.
- [ ] The tracked/default branch repo handles add, modify, delete, and rename
      changes and reaches `ready`.
- [ ] A wrong-branch repo fails closed without mutating the tracked index.
- [ ] A published artifact hydrates into a clean workspace and passes metadata,
      checksum, branch, commit, schema, and semantic-profile validation.
- [ ] Search or symbol lookup returns results that reflect the hydrated and
      reconciled file changes.
- [ ] Evidence records the exact supported deployment shape and remaining beta
      limitations.

**Scope notes**

This is the acceptance phase for the preceding contracts. Prefer a deterministic
local GitHub/CLI mock for normal CI and reserve live GitHub validation for an
operator evidence path if needed.

**Non-goals**

- No broad benchmark campaign.
- No new language support claim.
- No public release dispatch.

**Key files**

### Phase 6 — Multi-Repo Rollout Readiness (MRREADY)

**Objective**

Turn the hardening evidence into an operator-ready rollout gate for deploying
the stable package across many local repositories.

**Exit criteria**
- [ ] CLI/status output distinguishes `ready`, `local_only`, `publish_failed`,
      `wrong_branch`, `partial_index_failure`, and `index_unavailable` without
      relying on log inspection.
- [ ] Operator docs describe the safe deployment sequence for registering and
      validating many repositories on one machine.
- [ ] Runbooks explain recovery for failed publish, stale artifact, wrong
      branch, missing local index, and partial incremental failure.
- [ ] Public docs preserve the beta limitation around multi-repo and STDIO
      where it still applies.
- [ ] A final readiness note states whether the repo is ready for broad
      multi-repo deployment, controlled rollout only, or another hardening pass.

**Scope notes**

This phase should not add new core behavior unless a status/readiness gap from
MRE2E needs a narrow surfacing fix.

**Non-goals**

- No stable package release.
- No new topology support.
- No marketing launch assets.

**Key files**

## Execution Notes

- ARTPUB and IDXSAFE can be planned independently and executed in parallel if
  their owned files stay separated.
- WATCH should be planned after IDXSAFE freezes the mutation-result contract.
- CIFLOW should be planned after ARTPUB freezes the artifact asset contract.
- MRE2E should not execute until ARTPUB, IDXSAFE, WATCH, and CIFLOW are all
  complete, because it is the first full acceptance proof.
- MRREADY should remain docs/status-focused unless MRE2E exposes a narrow
  readiness-surfacing defect.
- This roadmap is hardening-only. If the work changes public release behavior,
  create a separate release-dispatch roadmap instead of expanding MRREADY.
